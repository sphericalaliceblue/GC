## 1. アプリ概要

LINEワークスに話しかけるだけでタスクを登録・管理できるシステム。
期限が近づくと自動でスマホに通知が届き、ボタン操作で進捗を返すと
スプレッドシートが自動更新される。

---

## 2. 全体の仕組み

```
【タスク登録】
  あなた（LINEワークス）
      ↓ 「来週金曜までに企画書作らなきゃ」
  GAS（受け取り窓口）
      ↓
  Dify（AIが内容を解析・整理）
      ↓ タスク名・期限・担当者を抽出
  GAS（確認メッセージを送信）← 登録前に内容を確認するステップ
      ↓
  あなた（LINEワークス）← 「登録する」or「期限を変更する」を選択
      ↓ 「登録する」をタップ
  GAS（スプレッドシートに保存）
      ↓
  あなた（LINEワークス）← 「登録しました！」と返信

【進捗通知】
  GAS（毎朝自動チェック）
      ↓ 期限が近いタスクを検出
  あなた（LINEワークス）← ボタン付き通知が3件届く
      ↓ ボタンをタップ
  GAS（スプレッドシートを自動更新）
```

---

## 3. タスク登録フロー（詳細）

```
ユーザーがテキストを送信
    ↓
[期限入力待ち状態？]──YES──→ 日付として解析 → 再確認メッセージ
    ↓ NO
Difyでタスク解析
    ↓
[タスク認識できた？]──NO──→ 「読み取れませんでした」を返信
    ↓ YES
キャッシュに一時保存（有効期限10分）
    ↓
確認メッセージを送信
「📋 タスク名  📅 期限  ／  登録する・期限を変更する」
    ↓
  ┌─「登録する」
  │      → スプレッドシートに正式登録
  │      → 完了メッセージ返信
  └─「期限を変更する」
         → 「新しい期限を入力してください」
         → ユーザーが日付を入力（YYYY-MM-DD）
         → 再度確認メッセージ表示
         → 「登録する」で正式登録
```

**注意：** 確認操作の有効期限は10分。期限切れの場合は再送信が必要。

---

## 4. 通知タイミングのルール

| 区分 | 期限まで | 通知するタイミング |
|------|---------|-----------------|
| 超短期 | 1日 | 当日の朝 |
| 短期 | 2〜5日 | 前日の朝 ＋ 当日の朝 |
| 中期・長期 | 6日以上 | 中間日の朝 ＋ 前日の朝 ＋ 当日の朝 |

**中間日の計算：** 登録日〜期限日の日数の半分の地点（切り上げ）

---

## 5. スプレッドシートの構成

シート名：`タスク一覧`

| 列 | 項目名 | 内容・形式 |
|----|--------|----------|
| A | タスクID | 自動採番（T001, T002…） |
| B | タスク名 | 例：企画書作成 |
| C | 期限 | YYYY-MM-DD 形式 |
| D | ステータス | 進行中 / 順調 / 完了 / 問題発生 / 中断 |
| E | 依頼者 | 例：ユーザー |
| F | 担当者 | LINEワークスのユーザーID |
| G | 作成日 | YYYY-MM-DD 形式 |
| H | 最終更新日 | YYYY-MM-DD 形式（自動） |
| I | 通知予定日リスト | JSON配列（自動計算） |
| J | 返信先ID | チャンネルIDまたはユーザーID |
| K | チャンネル宛 | TRUE（チャンネル宛）/ FALSE（DM宛） |
| L | 元文章 | 依頼者が送信した元のメッセージテキスト |

---

## 6. ステータスの定義と遷移

| ステータス | 意味 | 初期値 |
|-----------|------|:------:|
| 進行中 | タスク登録直後の状態 | ✅ |
| 順調 | 問題なく進んでいる | |
| 完了 | タスクが終わった | |
| 問題発生 | 何らかのトラブルが起きている | |
| 中断 | タスクを一時停止している | |

**通知スキップ：** `完了` と `中断` のタスクは期限通知をスキップする。

---

## 7. ボタンの種類と動作

通知は3つのメッセージに分かれて届く。

### メッセージ1：進捗ボタン
| ボタン | 動作 |
|--------|------|
| 順調 | ステータスを「順調」に更新 |
| 完了 | ステータスを「完了」に更新 |
| 問題発生 | ステータスを「問題発生」に更新 |
| 中断 | ステータスを「中断」に更新（以降の通知もスキップ） |

### メッセージ2：延期ボタン
| ボタン | 動作 |
|--------|------|
| 1日延期 | 期限を1日後ろにずらし、通知日も再計算 |
| 3日延期 | 期限を3日後ろにずらし、通知日も再計算 |
| 5日延期 | 期限を5日後ろにずらし、通知日も再計算 |

### メッセージ3：元文章確認
| ボタン | 動作 |
|--------|------|
| 元文章を見る | 依頼者が最初に送ったメッセージをテキストで返信 |

---

## 8. システムメッセージのフィルタ

LINEワークスはBotをチャンネルに追加した際に「利用開始」というテキストを自動送信する。
このメッセージはタスクとして登録されないよう、受信時に無視するフィルタを設けている。

---

## 9. グループチャンネル対応

| 送信元 | 動作 |
|--------|------|
| DM（個人トーク） | 全てのメッセージを処理する |
| グループチャンネル | Botへのメンション（`@Bot名`）がある場合のみ処理する |

返信先はメッセージ受信元（チャンネルまたはDM）に合わせて自動判定する。

---

## 10. 使用するサービスと役割

| サービス | 役割 |
|---------|------|
| GAS（Google Apps Script） | アプリ全体の司令塔。受け取り・保存・通知を担当 |
| Googleスプレッドシート | タスクデータの保管場所 |
| Dify | 自然な言葉をタスク情報（名前・期限・担当者）に変換 |
| LINEワークス Bot | スマホへの通知送信・ボタン操作の受け取り |
| CacheService（GAS） | タスク登録確認フローの一時データ保存（有効期限10分） |

## 11. ファイル構成

```

```

## 12. セットアップの流れ（概要）

1. Googleスプレッドシートを作成し、シートIDをメモ
2. LINEワークス DeveloperコンソールでBotを作成し、各種IDをメモ
3. Difyでワークフローを作成し、APIキーをメモ
4. GASプロジェクトを作成し、上記コードファイルを貼り付け
5. `Config.gs` に各IDを記入
6. GASを「ウェブアプリ」として公開し、URLをLINEワークスのCallback URLに設定
7. GASのトリガーで毎朝9時に `checkDeadlines` を実行するよう設定
8. スプレッドシートのヘッダー行（1行目）に列名を設定する

**スプレッドシートのヘッダー行（推奨）：**

```
タスクID | タスク名 | 期限 | ステータス | 依頼者 | 担当者 | 作成日 | 最終更新日 | 通知予定日 | 返信先ID | チャンネル宛 | 元文章
```
